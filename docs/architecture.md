# Architecture overview

This document summarises the major components and their boundaries. The
detailed Sui integration architecture, including state machines,
interfaces, and risk register, is in
[`sui-architecture-detail.md`](sui-architecture-detail.md).

## Layers

| Layer | Status | Responsibility |
|---|---|---|
| Cluster runtime | Implemented | Authenticated TCP / P2P transport for committee-scale (22-of-33 reference configuration) signing ceremonies. Round coordination, retry, timing, and node identity checks. |
| Signing core | Implemented | ML-DSA-65 threshold aggregation with Lagrange premultiplication. Per-node share loading and request-scoped one-time round state. |
| DKG / key generation | Implemented (testnet-grade) | Distributed key generation. Controlled-environment ceremony is acceptable for testnet-grade validation; audited DKG is a prerequisite for production custody. |
| Sui adapter | **Planned — next validation milestone** | Threshold Ed25519 / FROST backend producing standard Sui-compatible Ed25519 signatures. Sui sees one normal Ed25519 signer and one normal Sui address; Luvion enforces the 22-of-33 policy off-chain. The Ed25519 / FROST backend is part of the next validation milestone — see the [Sui integration status](sui-integration-status.md) for the 30–45 day plan. |
| Application layer | Optional | Move vault and similar asset-control modules (sits above account auth, not inside it). |

## Sui account-auth compatibility route (target design)

- 33 Luvion nodes hold threshold key shares generated by DKG or a
  controlled key ceremony; the ceremony keeps the full private key out
  of single-node memory, storage, and export flows.
- Any policy-approved set of 22 nodes cooperatively produces one
  standard 64-byte Ed25519 signature accepted by Sui's native ZIP215
  verifier.
- The current Ed25519 VRF / network key is retained for committee
  selection and network message authentication.
- A separate FROST aggregate key is planned for Sui account auth.

## Boundaries

- The Sui adapter is the **planned** near-term integration surface and
  is part of the next validation milestone, not a shipped component.
  ML-DSA-65 remains the long-term post-quantum migration layer and is
  not the Sui account signing mechanism.
- The Move vault is an application-layer module; it only controls
  assets deposited into it, and sits above account auth, not inside it.
- Sui protocol and Sui validator signature verification stay unchanged
  under this design.

## Original source documents

The full internal source documents — Sui Integration Technical
Architecture and Luvion Technical Summary, in PDF form — are released
under the **private review access process**, not as direct downloads in
this public repository. This is a deliberate choice: the PDFs in their
current form contain internal maintenance metadata (pinned dependency
versions, internal source paths, maintenance rules, export footers)
that the public-repo policy excludes. The
[sui-architecture-detail.md](sui-architecture-detail.md) document is
the public, sanitized derivative of the architecture PDF and contains
the same technical substance.

Reviewers who require the original PDFs in their unaltered form can
request them through the
[README contact path](../README.md#access-and-contact); they are
delivered alongside core-repo access.
