# Architecture overview

This document summarises the major components and their boundaries.

## Layers

| Layer | Responsibility |
|---|---|
| Cluster runtime | Authenticated TCP / P2P transport for committee-scale (22-of-33 reference configuration) signing ceremonies. Round coordination, retry, timing, and node identity checks. |
| Signing core | ML-DSA-65 threshold aggregation with Lagrange premultiplication. Per-node share loading and request-scoped one-time round state. |
| DKG / key generation | Distributed key generation. Controlled-environment ceremony is acceptable for testnet-grade validation; audited DKG is a prerequisite for production custody. |
| Sui adapter | Threshold Ed25519 / FROST backend producing standard Sui-compatible Ed25519 signatures. Sui sees one normal Ed25519 signer and one normal Sui address; Luvion enforces the 22-of-33 policy off-chain. |
| Application layer | Move vault and similar asset-control modules (sits above account auth, not inside it). |

## Sui account-auth compatibility route

- 33 Luvion nodes hold threshold key shares generated by DKG or a controlled
  key ceremony; the ceremony keeps the full private key out of single-node
  memory, storage, and export flows.
- Any policy-approved set of 22 nodes cooperatively produces one standard
  64-byte Ed25519 signature accepted by Sui's native ZIP215 verifier.
- The current Ed25519 VRF / network key is retained for committee selection
  and network message authentication.
- A separate FROST aggregate key is used for Sui account auth.

## Boundaries

- The Sui adapter is the near-term integration surface. ML-DSA-65 remains
  the long-term post-quantum migration layer and is not the Sui account
  signing mechanism.
- The Move vault is an application-layer module; it only controls assets
  deposited into it, and sits above account auth, not inside it.
- Sui protocol and Sui validator signature verification stay unchanged
  under this design.
