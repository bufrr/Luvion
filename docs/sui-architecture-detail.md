# Sui integration — technical architecture detail

This document is the public, sanitized technical detail behind the
[architecture overview](architecture.md) and the
[Sui integration status](sui-integration-status.md). It describes the
target design for the near-term Sui integration. The runtime and
implementation roadmap details that are not yet public are intentionally
omitted.

## 1. Technical conclusion

The near-term Sui integration targets **Sui's Ed25519
account-signature path** using **threshold Ed25519 / FROST**. Sui
supports multiple account-signature schemes (Ed25519, Secp256k1,
Secp256r1, multisig, zkLogin, passkey — see
[Sui transaction-auth signatures](https://docs.sui.io/concepts/cryptography/transaction-auth/signatures)).
The current core implements 22-of-33 threshold ML-DSA-65; Luvion keeps
threshold ML-DSA as the long-term post-quantum migration layer and adds
an Ed25519 / FROST transaction-signing backend on top of the existing
orchestration, network envelope, VRF proof, and cluster runtime to
produce a standard Ed25519 signature accepted by Sui under that
scheme.

Sui account-auth compatibility route:

- 33 Luvion nodes hold threshold key shares generated by DKG or a
  controlled key ceremony; the ceremony keeps the full private key out
  of single-node memory, storage, and export flows.
- Any policy-approved set of 22 nodes cooperatively produces one
  standard 64-byte Ed25519 signature accepted by the Sui / fastcrypto
  ZIP215 verifier.
- Sui sees one normal Ed25519 signer and one normal Sui address.
- Luvion enforces the 22-of-33 policy off-chain; Sui's native
  verification logic stays stable.

Supporting choices:

- Keep ML-DSA-65 as Luvion's long-term post-quantum migration layer;
  track threshold ML-DSA standardization maturity, third-party audit,
  and production hardening separately.
- Keep the current Ed25519 VRF / network key for committee selection
  and network message authentication.
- Use a new FROST aggregate key for Sui account-auth.
- Position Move shared-vault as an optional application-layer custody
  module that only controls assets deposited into the vault; Sui
  account-auth is handled by the aggregate Ed25519 address.

## 2. Goals and scope

Goals:

- Support 22-of-33 Luvion signing for Sui `TransactionData`.
- Output a Sui-native Ed25519 signature envelope:
  `0x00 || ed25519_signature || aggregate_public_key`.
- Integrate Sui object / gas resolution, dry-run / simulation, submit,
  and effects / checkpoint watching.
- Preserve a clear owned / shared object boundary: low-latency signing
  uses the address-owned execution path, public state transparency uses
  the shared-object path.
- Provide state machines, interfaces, test vectors, and failure-handling
  rules suitable for audit.

Scope boundaries:

- Sui protocol and Sui validator signature verification stay unchanged.
- Sui user-signature compatibility uses Ed25519 / FROST output.
- Luvion enforces the 22-of-33 policy off-chain.
- Optional Move vaults are application-layer custody modules.

## 3. Reusable capabilities

The current core repository already provides:

- 22-of-33 node orchestration, round coordination, retry, timing, and
  TCP cluster runtime.
- Per-node share loading, node identity checks, and request-scoped
  one-time round state.
- Network envelope signatures, VRF-style proofs, and base p2p /
  message routing modules.
- A complete experimental ML-DSA threshold signing path with
  rejection / retry operational experience.

Components to add for the Sui integration:

- An Ed25519 / FROST share format, separate from the current
  polynomial shares.
- FROST Round1 / Round2 wire DTOs for nonce commitments, partial
  signatures, partial verification, and blame records.
- A controlled key ceremony for testnet validation; audited DKG (or an
  independently reviewed secure key ceremony) is the production release
  gate.
- A threshold Ed25519 backend while keeping the current Ed25519 usage
  for network authentication and VRF proof.

## 4. Sui compatibility constraints

Sui Ed25519 user signature format:

```text
signature_bytes = 0x00 || ed25519_signature_64 || ed25519_public_key_32
```

Sui Ed25519 address derivation:

```text
sui_address = Blake2b256(0x00 || ed25519_public_key_32)
```

Sui transaction signing message:

```text
intent_message = IntentMessage<TransactionData> {
    intent: Intent::sui_transaction(),
    value: transaction_data,
}

message_to_sign = Blake2b256(BCS(intent_message))
```

Required rules:

- The signing input must be the intent digest Sui verifies for account
  signatures.
- FROST Round2 signs this 32-byte Blake2b256 digest as the Ed25519
  message bytes; Ed25519 / FROST-Ed25519 still performs its own
  internal SHA-512 computation.
- PTB bytes remain the transaction payload; the signing layer uses the
  intent digest.
- Transaction / effects digests remain object, effects, and query
  identities; user signatures use the `IntentMessage<TransactionData>`
  digest.
- `sender` must equal the Sui address derived from the aggregate
  public key.
- If `gas_owner != sender`, the sponsored transaction still needs gas
  owner authorization.
- Sui native account authentication follows Sui-supported signature
  schemes; Luvion uses Ed25519 / FROST to fit that boundary, while
  threshold ML-DSA advances as a separate long-term post-quantum
  migration layer.

## 6. Signer mode decision matrix

| Mode | 22-of-33 | Sui transaction auth | Status |
|---|---|---|---|
| Current ML-DSA-65 signer | Yes, inside Luvion | Kept as a separate backend | Continues serving the ML-DSA / PQ-ready path |
| Current Ed25519 usage | Single-signer | Network / VRF side | Continues serving VRF / network envelope |
| Sui native multisig | Up to 10 keys | Sui-native support | Reference only |
| Move shared-vault policy | Application-level policy | Gates shared-vault flows | Optional product-layer module |
| Threshold Ed25519 / FROST | Yes | Yes | **Planned** — Sui account-auth compatibility route |

The FROST backend must:

- Output a standard 64-byte Ed25519 signature.
- Output a 32-byte aggregate Ed25519 public key.
- Use RFC 9591 `FROST(Ed25519, SHA-512)` so the output satisfies the
  cofactored verification equation accepted by Sui / fastcrypto ZIP215.
- Verify FROST output against Sui fastcrypto ZIP215 test vectors before
  making a deployment-readiness claim.
- Use the same fastcrypto / ZIP215 verification mode for local
  self-verification.
- Bind each session to `request_id`, network, intent digest, aggregate
  public key, participant set, and nonce commitments.
- Verify partial signatures before aggregation and produce blame
  records on failure.
- Use per-session nonces; retries must use fresh nonces.

## 7. Signing state machine

```text
Draft
  -> ObjectsResolved
  -> Simulated
  -> PolicyApproved
  -> FrostRound1
  -> FrostRound2
  -> SignatureAggregated
  -> Submitted
  -> Checkpointed
```

| State | Description |
|---|---|
| `Draft` | Intent, operations, objects, and policy context are being assembled |
| `ObjectsResolved` | Object refs, gas coin, gas price / budget are resolved |
| `Simulated` | Dry-run / dev-inspect returned expected effects and gas estimate |
| `PolicyApproved` | Off-chain policy engine approved signing |
| `FrostRound1` | Nodes submit nonce commitments |
| `FrostRound2` | Nodes submit partial signatures |
| `SignatureAggregated` | Sui Ed25519 signature is aggregated and locally verified |
| `Submitted` | Transaction submitted to Sui fullnode / RPC |
| `Checkpointed` | Effects confirmed and available for audit |

### Exception handling

| Exception | Transition | Handling |
|---|---|---|
| stale object ref | `ObjectsResolved -> Draft` | Re-fetch object version / digest and rebuild transaction |
| insufficient gas budget | `Simulated -> Draft` | Adjust gas budget / payment and simulate again |
| policy rejection | `PolicyApproved -> Rejected` | Record rejection reason and skip FROST |
| Round1 timeout | `FrostRound1 -> Retry` | Select a new session and regenerate nonce commitments |
| Round2 timeout | `FrostRound2 -> Retry` | Discard session; use fresh nonce |
| invalid partial | `FrostRound2 -> FailedFinal` or replacement node | Create blame record and isolate node by policy |
| duplicate commitment | `FrostRound1 -> FailedFinal` | Treat as protocol violation |
| nonce reuse | `FrostRound* -> FailedFinal` | Treat as critical security event |
| Sui submit reject | `Submitted -> Draft` | Check signature, object refs, epoch, gas, and sender |
| Move abort | `Submitted -> FailedFinal` | Record effects and abort location |
| object lock / equivocation | `Submitted -> OperationalIncident` | Epoch-scoped operational incident; after manual resolution, re-pin object refs and return to `Draft` |

### Mandatory checks

- Signing layer uses the intent digest.
- Recompute the intent digest before signing and before submit.
- Verify `sender == SuiAddress::from(aggregate_public_key)`.
- Use fresh nonce material for every retry.
- Verify every partial signature before aggregation.
- Locally verify the final Ed25519 signature before submit.

## 10. Verification checklist

Cryptography:

- FROST aggregate signature is accepted by Sui fastcrypto ZIP215
  Ed25519 verifier.
- FROST aggregate signature passes fixed Sui fastcrypto ZIP215 test
  vectors.
- Any invalid partial is detected and attributed.
- Retry uses fresh nonce material.
- Participant ordering is deterministic.
- Aggregate public key and Sui address derivation match.

Sui compatibility:

- Intent digest matches Sui SDK / source test vectors.
- `tx_bytes` matches `TransactionData` BCS encoding.
- Signature envelope format is `0x00 || signature || public_key`.
- Sender / gas owner / sponsor rules are covered.
- Stale object refs trigger transaction rebuild.

System reliability:

- Signature production requires quorum.
- Node timeout retry uses a new session.
- Blame records are auditable.
- Submit failure and effects failure are recorded separately.
- Checkpoint watcher handles delayed confirmation and query failure.

## 11. Main technical risks

| Risk | Impact | Control |
|---|---|---|
| nonce reuse | Aggregate private key recovery and full key compromise | Session-level nonce lifecycle, persistent replay guard, fresh nonce on retry, critical security alerting |
| FROST ciphersuite semantic drift from Sui Ed25519 verifier | Signature submission failure | Use RFC 9591 `FROST(Ed25519, SHA-512)` and align local / on-chain verification on Sui fastcrypto ZIP215 semantics |
| DKG / import ceremony audit gap | Insufficient key-material production assurance | RFC 9591 trusted-dealer keygen baseline; testnet may use controlled import ceremony; production release gate is audited DKG or independently reviewed secure key ceremony |
| ZIP215 / RFC 8032 local verifier drift | Local self-verification and on-chain verification diverge | Pin local self-verification to Sui fastcrypto verifier / mode |
| threshold ML-DSA standardization maturity | PQ backend production timing depends on standards and audit | Sui account-auth compatibility uses Ed25519 / FROST; threshold ML-DSA continues as the long-term post-quantum migration layer |
| stale object refs | Submit rejection or object lock conflict | Recompute and rebuild across resolve / simulate / sign / submit boundaries |
| missing sponsored tx authorization | Signature verification failure | Separate sender signature and gas owner signature responsibilities |
| shared object contention / object lock / stale reference | Higher latency, lock conflict, or submission failure | Default to address-owned; benchmark shared-path contention, Move policy gas, and object lock / stale reference failure modes |
| Move vault / account-auth boundary confusion | Asset-control semantics are classified incorrectly | Keep account-auth and application-level vault paths explicitly separate |
